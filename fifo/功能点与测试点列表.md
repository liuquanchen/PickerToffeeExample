# SyncFIFO 功能覆盖率验证计划

## 1. 文档目的

本文档基于当前 RTL 实现 SyncFIFO.v 对功能点进行重构，并针对每个功能点拆解测试点、输入激励条件、预期 RTL 行为以及需要观测的信号，用于指导功能覆盖率验证与测试用例设计。

本文档只描述当前 RTL 的实际行为，不扩展到理想 FIFO 规格。

## 2. RTL 摘要

根据 SyncFIFO.v 的实现，当前模块具备如下特征：

- 单时钟同步 FIFO，时钟为 `clk`。
- 低有效同步复位，复位信号为 `rst_n`。
- 数据位宽为 32 bit，FIFO 深度为 16。
- FIFO 满条件由 `counter == 16` 决定，空条件由 `counter == 0` 决定。
- 写请求生效条件为 `wvalid = we_i && !full_o`。
- 读请求生效条件为 `rvalid = re_i && !empty_o`。
- `wptr`、`rptr` 为 4 bit 指针，在有效写/读时更新。
- `counter` 为 5 bit 计数器，在“仅写”或“仅读”时更新，在“同时有效读写”或“均无效”时保持不变。
- `data_o` 仅在复位或有效读时更新。

## 3. 重构后的 RTL 功能点列表

### 3.1 复位行为

- 在时钟上升沿采到 `rst_n = 0` 时，`wptr`、`rptr`、`data_o`、`counter` 被清零。
- 复位后，`empty_o = 1`，`full_o = 0`。

### 3.2 读/写通路行为

- 当 `we_i = 1` 且 FIFO 非满时，写请求生效。生效写请求将 `data_i` 写入当前 `wptr` 指向的存储单元。
- 当 `re_i = 1` 且 FIFO 非空时，读请求生效。生效读请求将当前 `rptr` 指向存储单元的数据送到 `data_o`。
- 在 FIFO 既非空也非满时，允许同周期同时读写。
- 在 FIFO 空状态下同时给出读写请求时，实际只会发生写。
- 在 FIFO 满状态下同时给出读写请求时，实际只会发生读。
- 当 FIFO 已满时，写请求被阻塞。
- 当 FIFO 为空时，读请求被阻塞。

### 3.3 读/写指针更新行为

- `wptr` 只在有效写时更新，并在时钟上升沿加 1。
- `wptr` 为 4 bit，连续有效写后会发生**回绕**。
- `rptr` 只在有效读时更新，并在时钟上升沿加 1。
- `rptr` 为 4 bit，连续有效读后会发生**回绕**。
- 写指针追上读指针（写满），读指针追上写指针（读空）。

### 3.6 计数器更新行为

- 仅写时，`counter` 加 1。
- 仅读时，`counter` 减 1。
- 同时有效读写时，`counter` 保持不变。
- 无有效读写时，`counter` 保持不变。

### 3.7 状态输出

- `counter == 0` 时，`empty_o = 1`。
- `counter == 16` 时，`full_o = 1`。

## 4. 功能覆盖率验证计划

信号观测与覆盖思路
- 对 `wptr`、`rptr`、`counter` 的覆盖确认应采用**时钟上升沿前后取样**的方式，检查是否发生了预期更新。
- 对写入行为，建议记录写入前的 `wptr`，并在写入的**下一拍**检查 `data_o` 是否等于写入的 `data_i`。

### 4.1 复位行为

| 测试点 ID | 输入激励条件 | 预期 RTL 行为 | 需观测信号 | 覆盖确认条件 |
|---|---|---|---|---|
| RST-01 | 经过若干次读写后，在时钟上升沿令 `rst_n = 0` | `wptr = 0`，`rptr = 0`，`data_o = 0`，`counter = 0` | `rst_n`、`wptr`、`rptr`、`data_o`、`counter` | 命中“非缺省状态下进入复位”，且上述寄存器在该上升沿后全部清零 |
| RST-02 | `rst_n = 0` 持续多个时钟周期 | 状态持续保持在复位值 | `rst_n`、`wptr`、`rptr`、`data_o`、`counter` | 连续多个周期采样到状态不变化，始终保持复位值 |
| RST-03 | 复位释放后保持 `we_i = 0`、`re_i = 0` | 无自发读写，状态维持复位后的稳定值 | `rst_n`、`we_i`、`re_i`、`wptr`、`rptr`、`counter`、`data_o` | 复位释放后的空闲周期中，状态保持不变 |

### 4.2 写通路行为

| 测试点 ID | 输入激励条件 | 预期 RTL 行为 | 需观测信号 | 覆盖确认条件 |
|---|---|---|---|---|
| WR-01 | `we_i = 1`，`re_i = 0`，且 FIFO 非满 | `wvalid = 1`，`data_i` 被写入当前 `wptr` 指向单元 | `we_i`、`full_o`、`wvalid`、`data_i`、`ram[]`、`wptr`、`counter` | 采样到 `wvalid = 1`，并确认 `ram[old_wptr] == data_i` |
| WR-02 | 连续多个周期执行有效写，FIFO 始终非满 | 每个周期都成功写入，写入数据按接受顺序进入存储阵列 | `we_i`、`wvalid`、`data_i`、`ram[]`、`wptr`、`counter` | 命中多拍连续有效写，且每拍都能确认写入成功 |
| WR-03 | 当前 `counter = 15` 时施加有效写 | 本拍写入成功，写后 `counter = 16`，FIFO 进入满状态 | `wvalid`、`counter`、`full_o`、`wptr` | 命中 `counter: 15 -> 16` 转移，且 `full_o` 置 1 |
| WR-04 | `we_i = 1`，且 FIFO 已满 | 写请求被阻塞，不发生写入 | `we_i`、`full_o`、`wvalid`、`wptr`、`counter`、`ram[]` | 采样到 `wvalid = 0`，同时 `wptr`、`counter`、目标存储单元均不变化 |

### 4.3 读通路行为

| 测试点 ID | 输入激励条件 | 预期 RTL 行为 | 需观测信号 | 覆盖确认条件 |
|---|---|---|---|---|
| RD-01 | `re_i = 1`，`we_i = 0`，且 FIFO 非空 | `rvalid = 1`，当前 `rptr` 指向的数据送到 `data_o` | `re_i`、`empty_o`、`rvalid`、`data_o`、`ram[]`、`rptr`、`counter` | 采样到 `rvalid = 1`，且 `data_o` 等于 `ram[old_rptr]` |
| RD-02 | 连续多个周期执行有效读，FIFO 始终非空 | 每个周期都成功读出，`data_o` 输出顺序与接受写入顺序一致 | `re_i`、`rvalid`、`data_o`、`rptr`、`counter` | 命中多拍连续有效读，并通过参考队列确认输出顺序正确 |
| RD-03 | 当前 `counter = 1` 时施加有效读 | 本拍读出成功，读后 `counter = 0`，FIFO 进入空状态 | `rvalid`、`counter`、`empty_o`、`rptr`、`data_o` | 命中 `counter: 1 -> 0` 转移，且 `empty_o` 置 1 |
| RD-04 | `re_i = 1`，且 FIFO 为空 | 读请求被阻塞，`data_o`、`rptr`、`counter` 保持不变 | `re_i`、`empty_o`、`rvalid`、`data_o`、`rptr`、`counter` | 采样到 `rvalid = 0`，同时 `data_o`、`rptr`、`counter` 均不变化 |
| RD-05 | 先写入若干数据，再部分读出，再继续写入至指针回绕，最后完全读空 | 读出顺序仍与实际接受写入顺序一致 | `data_i`、`data_o`、`wptr`、`rptr`、`counter` | 命中跨回绕场景，并通过参考队列确认 FIFO 顺序保持正确 |

### 4.4 写指针更新行为

| 测试点 ID | 输入激励条件 | 预期 RTL 行为 | 需观测信号 | 覆盖确认条件 |
|---|---|---|---|---|
| WPTR-01 | 执行一次有效写 | `wptr` 在该时钟上升沿后加 1 | `wvalid`、`wptr` | 命中 `wvalid = 1`，并观测到 `wptr: N -> N+1` |
| WPTR-02 | `we_i = 0`，或 `we_i = 1` 但 FIFO 已满 | `wptr` 保持不变 | `we_i`、`full_o`、`wvalid`、`wptr` | 分别命中“无写请求”和“满时写阻塞”两类场景，且 `wptr` 均保持稳定 |
| WPTR-03 | 执行足够多次有效写，使 `wptr` 从 15 再次加 1 | `wptr` 发生 4 bit 回绕 | `wvalid`、`wptr`、`counter` | 命中 `wptr: 15 -> 0` 转移 |

|---|---|---|---|---|
| RPTR-01 | 执行一次有效读 | `rptr` 在该时钟上升沿后加 1 | `rvalid`、`rptr` | 命中 `rvalid = 1`，并观测到 `rptr: N -> N+1` |
| RPTR-02 | `re_i = 0`，或 `re_i = 1` 但 FIFO 为空 | `rptr` 保持不变 | `re_i`、`empty_o`、`rvalid`、`rptr` | 分别命中“无读请求”和“空时读阻塞”两类场景，且 `rptr` 均保持稳定 |
| RPTR-03 | 执行足够多次有效读，使 `rptr` 从 15 再次加 1 | `rptr` 发生 4 bit 回绕 | `rvalid`、`rptr`、`counter` | 命中 `rptr: 15 -> 0` 转移 |

### 4.6 计数器更新行为

| 测试点 ID | 输入激励条件 | 预期 RTL 行为 | 需观测信号 | 覆盖确认条件 |
|---|---|---|---|---|
| CNT-01 | `we_i = 1`、`re_i = 0`，且 FIFO 非满 | 仅写有效，`counter` 加 1 | `wvalid`、`rvalid`、`counter` | 命中 `wvalid = 1 && rvalid = 0`，并观测到 `counter: N -> N+1` |
| CNT-02 | `we_i = 0`、`re_i = 1`，且 FIFO 非空 | 仅读有效，`counter` 减 1 | `wvalid`、`rvalid`、`counter` | 命中 `wvalid = 0 && rvalid = 1`，并观测到 `counter: N -> N-1` |
| CNT-03 | `we_i = 1`、`re_i = 1`，且 FIFO 既非空也非满 | 同时有效读写，`counter` 保持不变 | `wvalid`、`rvalid`、`counter`、`wptr`、`rptr` | 命中 `wvalid = 1 && rvalid = 1`，同时确认 `counter` 稳定 |
| CNT-04 | `we_i = 0`、`re_i = 0`，或请求存在但均未生效 | 无有效读写，`counter` 保持不变 | `wvalid`、`rvalid`、`counter` | 命中 `wvalid = 0 && rvalid = 0`，且 `counter` 稳定 |

### 4.7 状态输出与并发行为

| 测试点 ID | 输入激励条件 | 预期 RTL 行为 | 需观测信号 | 覆盖确认条件 |
|---|---|---|---|---|
| STAT-01 | 使 `counter = 0` | `empty_o = 1`，`full_o = 0` | `counter`、`empty_o`、`full_o` | 命中 `counter == 0`，且 `empty_o == 1 && full_o == 0` |
| STAT-02 | 使 `counter` 从 0 变为 1，再从 1 变为 0 | `empty_o` 随 `counter` 在空/非空之间切换 | `counter`、`empty_o` | 同时命中 `0 -> 1` 与 `1 -> 0` 两类转移 |
| STAT-03 | 使 `counter = 16` | `full_o = 1`，`empty_o = 0` | `counter`、`full_o`、`empty_o` | 命中 `counter == 16`，且 `full_o == 1 && empty_o == 0` |
| STAT-04 | 使 `counter` 从 15 变为 16，再从 16 变为 15 | `full_o` 随 `counter` 在非满/满之间切换 | `counter`、`full_o` | 同时命中 `15 -> 16` 与 `16 -> 15` 两类转移 |
| CONC-01 | `we_i = 1`、`re_i = 1`，且 `0 < counter < 16` | 同周期同时读写，`wvalid = 1`，`rvalid = 1`，`wptr`/`rptr` 均更新，`counter` 不变 | `we_i`、`re_i`、`wvalid`、`rvalid`、`wptr`、`rptr`、`counter` | 命中 `wvalid = 1 && rvalid = 1`，并确认双指针更新、计数器稳定 |
| CONC-02 | `we_i = 1`、`re_i = 1`，且 `counter = 0` | 仅写生效，`wvalid = 1`，`rvalid = 0`，行为等效于写操作 | `we_i`、`re_i`、`wvalid`、`rvalid`、`wptr`、`rptr`、`counter` | 命中空状态下的同时请求，并确认只发生写通路更新 |
| CONC-03 | `we_i = 1`、`re_i = 1`，且 `counter = 16` | 仅读生效，`wvalid = 0`，`rvalid = 1`，行为等效于读操作 | `we_i`、`re_i`、`wvalid`、`rvalid`、`wptr`、`rptr`、`counter` | 命中满状态下的同时请求，并确认只发生读通路更新 |

## 6. 覆盖建模建议

为便于后续构建功能覆盖模型，建议至少包含以下覆盖维度：

1. **状态覆盖**
   - `counter` 关键状态：`0`、`1`、`15`、`16`
   - `wptr`、`rptr` 的普通递增与回绕状态

2. **行为覆盖**
   - `wvalid` / `rvalid` 四种组合：
     - `00`：均无效
     - `10`：仅写有效
     - `01`：仅读有效
     - `11`：同时有效读写

3. **转移覆盖**
   - `counter` 的关键转移：`0 -> 1`、`1 -> 0`、`15 -> 16`、`16 -> 15`
   - `wptr`、`rptr` 的 `15 -> 0` 回绕转移

4. **数据顺序覆盖**
   - 连续写连续读场景
   - 交错读写场景
   - 指针回绕后的数据顺序场景

## 7. 结论

基于当前 RTL，功能覆盖率验证的核心不只是覆盖“写、读、满、空”这几个表面现象，还需要同时覆盖：

- 请求是否真正生效（`wvalid` / `rvalid`）
- 指针是否在正确条件下更新
- 计数器是否在不同读写组合下正确变化
- 满/空标志是否与 `counter` 状态严格一致
- 数据在连续访问、交错访问、回绕访问下是否保持正确顺序

按照本文档执行后，可以形成一套结构清晰、覆盖目标明确的 SyncFIFO 功能覆盖率验证计划。
